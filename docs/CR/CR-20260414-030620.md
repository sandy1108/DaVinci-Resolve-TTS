# DaVinci Resolve TTS Code Review

- 审查时间：2026-04-14 03:06:20（UTC+8）
- 审查范围：`DaVinci Resolve TTS/DaVinci TTS mod by WSGH.py` 及仓库内现有说明文档
- 审查方式：静态代码审查，未在 DaVinci Resolve 运行时做实机回归

## 主要问题

### 1. 高：依赖导入失败后脚本仍继续初始化，最终会在模块加载阶段直接崩溃

- 位置：`DaVinci Resolve TTS/DaVinci TTS mod by WSGH.py:239-288`、`DaVinci Resolve TTS/DaVinci TTS mod by WSGH.py:293-301`、`DaVinci Resolve TTS/DaVinci TTS mod by WSGH.py:2129-2136`
- 问题：
  - 依赖首次导入失败后，脚本会尝试从 `Lib` 目录再次导入。
  - 如果第二次导入仍失败，代码只打印错误，不中断初始化。
  - 但后续马上无条件执行 `requests.Session()`、`HTTPAdapter(...)`，并在更后面无条件访问 `speechsdk.SpeechSynthesisOutputFormat`。
- 影响：
  - 只要 `requests`、`azure.cognitiveservices.speech`、`edge_tts` 中任意关键依赖仍未就绪，脚本就不是“带提示降级”，而是直接在启动阶段抛 `NameError` / `AttributeError`。
  - 这会让“缺依赖”变成“插件完全打不开”，用户也拿不到清晰的修复指引。
- 建议：
  - 第二次导入失败后立即停止初始化，并弹出明确错误提示。
  - 把依赖能力拆成显式布尔标记，例如 `HAS_REQUESTS`、`HAS_SPEECHSDK`，后续功能按能力启停，不要无条件引用未导入对象。

### 2. 高：三个“朗读当前字幕”入口都没有处理“播放头下没有字幕”的情况，容易直接抛异常

- 位置：`DaVinci Resolve TTS/DaVinci TTS mod by WSGH.py:2683-2709`、`DaVinci Resolve TTS/DaVinci TTS mod by WSGH.py:2748-2753`、`DaVinci Resolve TTS/DaVinci TTS mod by WSGH.py:3330-3345`、`DaVinci Resolve TTS/DaVinci TTS mod by WSGH.py:3446-3459`
- 问题：
  - `get_current_subtitle()` 在未命中字幕时返回 `(None, current_frame, current_frame)`。
  - Azure“朗读当前字幕”随后把 `subtitle` 直接传给 `generate_filename()`，该函数会调用 `subtitle.replace(...)`。
  - MiniMax / OpenAI 的“朗读当前字幕”同样直接沿用这份返回值，没有做空值兜底。
- 影响：
  - 用户只要把播放头放在非字幕区域点击“朗读当前字幕”，就可能直接异常退出当前流程，而不是收到“当前位置没有字幕”的友好提示。
  - 这是核心主路径问题，命中概率高。
- 建议：
  - 在三个入口统一判断 `text is None or not text.strip()`，提前提示并返回。
  - 最好把这层校验下沉成公共函数，避免三个 Provider 分支重复踩坑。

### 3. 高：MiniMax 生成字幕时没有把音频插入位置传给字幕导入逻辑，字幕会回到时间线开头

- 位置：`DaVinci Resolve TTS/DaVinci TTS mod by WSGH.py:985-1066`、`DaVinci Resolve TTS/DaVinci TTS mod by WSGH.py:3293-3315`
- 问题：
  - `process_minimax_request()` 已经通过 `timeline_func()` 计算出了 `start_frame` / `end_frame`，并用它把音频放到正确位置。
  - 但处理字幕时调用的是 `import_srt_to_timeline(srt_path)`，没有传入 `start_frame`。
  - `import_srt_to_timeline()` 的偏移逻辑完全依赖 `insert_position_frame` 参数。
- 影响：
  - 文本框发起的 MiniMax 合成会出现“音频插在当前播放头，字幕却从 0 秒开始”的错位。
  - 更严重的是，增量替换逻辑会基于错误时间段删除旧字幕，导致时间线前部字幕被误删。
- 建议：
  - 这里应改为传入同一次合成得到的 `start_frame`。
  - 同时建议给该路径补一个最小回归用例：时间线中段插入文本，校验音频与字幕起点一致。

### 4. 中：克隆音频渲染会修改项目渲染模式/预设，但没有恢复现场

- 位置：`DaVinci Resolve TTS/DaVinci TTS mod by WSGH.py:706-776`
- 问题：
  - `render_audio_by_marker()` 会直接执行 `SetCurrentRenderMode(1)` 和 `LoadRenderPreset(...)`。
  - 不论渲染成功、失败还是中途提前返回，函数都没有把原来的渲染模式和预设还原。
- 影响：
  - 插件会把 Resolve 项目的全局渲染状态留在“音频导出”上下文，污染用户后续手工渲染操作。
  - 这是典型的宿主状态泄漏，尤其在编辑软件插件里比较危险。
- 建议：
  - 进入函数时保存原始渲染模式和预设，`try/finally` 中统一恢复。
  - 早返回分支也应走同一套恢复逻辑。

### 5. 中：加载窗口用后台线程直接更新 Fusion UI，和仓库文档里的线程约束相冲突

- 位置：`DaVinci Resolve TTS/DaVinci TTS mod by WSGH.py:205-216`
- 参考：`docs/DaVinci_Resolve_集成原理讲解.md`
- 问题：
  - `_loading_timer_worker()` 在后台线程里直接写 `_loading_items["LoadLabel"].Text`。
  - 仓库文档已经明确写到 UI 更新应在主线程中进行。
- 影响：
  - 在宿主 UI 框架下，这类跨线程改 UI 的代码很容易造成偶发卡死、闪退或界面状态不同步。
  - 问题可能不会每次复现，但一旦复现很难定位。
- 建议：
  - 改成主线程定时器或事件派发方式更新文案。
  - 至少不要在工作线程中直接写控件属性。

### 6. 中：部分 Azure 设置只能读取不能保存，重启后会悄悄丢失

- 位置：`DaVinci Resolve TTS/DaVinci TTS mod by WSGH.py:117-159`、`DaVinci Resolve TTS/DaVinci TTS mod by WSGH.py:2155-2166`、`DaVinci Resolve TTS/DaVinci TTS mod by WSGH.py:3510-3554`
- 问题：
  - 默认配置里定义了 `STYLE`、`BREAKTIME`。
  - 启动时会尝试读取 `STYLE`。
  - 但关闭保存时没有写回 `STYLE` 和 `BREAKTIME`。
- 影响：
  - 用户每次关闭插件后，Azure 风格和停顿时长会回退为默认值。
  - 这类“静默丢设置”很影响可用性，也会让用户误以为配置文件损坏。
- 建议：
  - 保存、加载、重置三处字段保持完全对齐。
  - 最好把配置字段集中定义，避免手工维护多份键名清单。

## 开放问题与推测

- 推测：`connect_resolve()` 依赖全局 `resolve` 对象，但文件中没有看到 `resolve = GetResolve()` 的显式赋值，位置在 `DaVinci Resolve TTS/DaVinci TTS mod by WSGH.py:667-674`。如果当前运行环境始终由 Resolve 注入全局 `resolve`，那这里不会出问题；如果存在外部 Python 启动或某些宿主差异，这里会有兼容性风险。这个点我没有在本地 Resolve 环境里实机验证，所以先标为推测而不是已确认缺陷。

## 测试缺口

- 仓库里没有看到自动化测试或最小回归脚本，当前高风险路径主要靠手工点击验证。
- 建议至少补 4 个回归场景：
  - 播放头下无字幕时点击三个“朗读当前字幕”入口。
  - MiniMax 文本框合成并同步字幕到非 0 起点。
  - 克隆渲染前后检查 Resolve 渲染模式是否恢复。
  - 缺失依赖时插件能否给出可理解的错误提示，而不是直接启动失败。

## 结论

当前脚本已经覆盖了比较完整的业务链路，但主路径上仍有几处会直接影响用户工作的稳定性问题，优先级最高的是：

1. 依赖导入失败后的崩溃路径。
2. “朗读当前字幕”在无字幕场景下的空值崩溃。
3. MiniMax 字幕导入未传插入位置导致的音频/字幕错位与误删。

这三项建议优先修复，再进入体验层面的细化优化。
