## 1. 问题全景

在这个项目里，字幕相关问题并不是单一 bug，而是一组互相关联的问题：

1. 音频插入到正确位置，但字幕插入到错误位置。
2. 字幕整体前移或后移几秒，和音频不同步。
3. 字幕第一次导入位置不对，二次校正后又截断了头部。
4. 新字幕导入后，旧字幕被误删，甚至同一次新导入的后半段字幕也被删掉。
5. 某些字幕插入方案在 Resolve 中会直接触发闪退。

这些问题之所以容易被误判，是因为它们表面都表现为“字幕不对”，但底层成因并不相同。

---

## 2. 典型现象归类

### 2.1 插入位置不对

典型表现：

- 音频在目标播放头位置插入成功。
- 字幕却没有从同一个时间点开始。
- 有时字幕会整体偏后，有时会整体偏前。

### 2.2 时间偏移被错误处理

典型表现：

- MiniMax 返回的字幕时间从 `00:00:00,000` 开始。
- 代码试图按“音频目标起点”整体平移字幕时间。
- 在某些场景下，二次平移导致 Resolve 再次错误解释或把字幕头部截断。

### 2.3 误删旧字幕

典型表现：

- 新字幕导入后，目标区间前后的旧字幕被删除。
- 更糟的是，新导入字幕的后半段也可能被误删。
- 使用 `Ctrl + Z` 可以看到被删掉的是已有字幕，而不是简单的显示问题。

### 2.4 导入时闪退

典型表现：

- 音频先插入成功。
- 字幕导入阶段 Resolve 直接闪退。
- 问题常发生在尝试使用字幕 `clipInfo` 精确插入时。

---

## 3. 原始分析里哪些结论是对的，哪些后来被推翻了

### 3.1 仍然成立的结论

以下结论在后续实测中仍成立：

- MiniMax 返回的字幕时间是“相对于音频文件本身”的时间。
- 音频插入和字幕插入在 Resolve 脚本 API 中的行为并不完全一致。
- 如果在未确认真实落点前删除旧字幕，风险非常高。
- 需要保留“增量替换”能力，而不是每次清空所有字幕轨道。

### 3.2 已被推翻或修正的结论

以下结论在后续实测中被证明不可靠：

- “字幕应优先使用 `clipInfo + recordFrame` 精确插入。”
  原因：在当前项目与 Resolve `20.2.1` 组合下，这条路径曾触发闪退。

- “字幕时间只要加上 `insert_position_frame` 就一定正确。”
  原因：Resolve 对字幕导入的真实行为不只是简单的“按 SRT 时间戳放置”。

- “`AppendToTimeline()` 的返回值可以直接视为本次导入的全部新字幕项。”
  原因：实测发现它可能只返回一部分新字幕，不能稳定覆盖全集。

- “只要按预期时间范围删除旧字幕，就不会误删。”
  原因：一旦字幕真实落点和预期不一致，就会把用户已有字幕甚至本次新字幕删掉。

## 4. Resolve API 行为差异：音频为什么稳定，字幕为什么不稳定

### 4.1 音频插入路径

项目里的音频插入核心是：

```python
clip_info = {
    "mediaPoolItem": selected_clip,
    "startFrame": 0,
    "endFrame": clip_duration_frames - 1,
    "trackIndex": track_index,
    "recordFrame": start_frame,
    "stereoEye": "both"
}

media_pool.AppendToTimeline([clip_info])
```

这条路径在当前项目里是稳定的，原因有两个：

- 音频素材使用的是明确的 `clipInfo`。
- `recordFrame` 在音频场景下能稳定表达“时间线上的插入位置”。

### 4.2 字幕插入路径

字幕当前采用的是：

```python
imported = media_pool.ImportMedia([srt_path])
new_clip = imported[0]
media_pool.AppendToTimeline([new_clip])
```

这意味着字幕走的是“导入字幕媒体，再把媒体追加到时间线”的路径，而不是音频那种稳定的 `clipInfo` 路径。

### 4.3 为什么不能直接把字幕改成和音频一样

历史上曾尝试过这样的方向：

```python
clip_info = {
    "mediaPoolItem": subtitle_clip,
    "startFrame": 0,
    "endFrame": subtitle_duration_frames - 1,
    "trackIndex": track_index,
    "recordFrame": start_frame,
}
```

从静态推理看，这很合理；但在当前项目的实测里，这条路径并不稳定，且曾触发 Resolve 闪退。因此：

- 这条方案可以保留为“历史尝试”；
- 但不能再被写成当前项目的默认修复方案。

---

## 5. 时间偏移问题的真实样子

### 5.1 从业务角度看，偏移逻辑为什么看起来合理

如果音频在时间线 `01:02:15:02` 插入，而 MiniMax 返回字幕时间从 `00:00:00,000` 开始，那么直觉上会得到：

- 音频目标起点 = `start_frame`
- 字幕每条时间 = 原始字幕时间 + `start_frame`

这在纸面上完全说得通。

### 5.2 但实测表明，字幕导入并不是纯数学偏移问题

后续日志说明：

- Resolve 的字幕最终落点不仅取决于 SRT 时间戳；
- 还会受到当前播放头位置和宿主内部导入行为影响；
- 在音频插入后，播放头位置可能被推进；
- 如果字幕导入直接沿用这个新播放头位置，就会整体错位。

因此，字幕导入前必须显式把播放头重新设回音频目标起点，而不能假设“当前播放头还在原位置”。

---

## 6. 为什么直接改写 SRT 时间戳不是通用解

### 6.1 正向补偿时，它是可用工具

如果实测发现字幕整体偏后，需要把字幕整体再往后推，生成一个校正副本，例如 `_positioned.srt`，是可行的：

- 原始 `.srt` 保留 MiniMax 的原始结果；
- `_positioned.srt` 作为位置校正副本；
- 只在确实需要正向补偿时使用。

### 6.2 负向补偿时，它会暴露格式边界

如果实测发现字幕整体偏后很多，脚本可能会计算出一个负偏移量，例如：

- 目标起点：`123181`
- 实际起点：`123747`
- 需要补偿：`-566` 帧

这时如果把整份 SRT 直接整体前移：

- 头部字幕时间会跌到 `00:00:00,000` 之前；
- `SRT` 本身不能表达负时间戳；
- 结果就是字幕头部被截断、折叠或消失。

因此：

- 正向补偿可以用 `_positioned.srt`；
- 负向补偿不能简单靠生成“负时间戳 SRT”解决。

---

## 7. 闪退问题的真实结论

### 7.1 历史分析为什么会认为 `clipInfo` 是正确方向

早期分析的依据是：

- 音频使用 `clipInfo + recordFrame + trackIndex` 很稳定；
- 字幕使用同样模式，理论上也应该能精确插入；
- 如果补齐 `trackIndex`、修正 `endFrame`，似乎应当能解决问题。

这个分析过程本身并不离谱。

### 7.2 但当前环境下实测结果否定了它

在当前项目与 Resolve `20.2.1` 组合下，字幕 `clipInfo` 路线实测曾造成：

- 音频已插入；
- 字幕导入阶段崩溃；
- Resolve 直接闪退。

因此当前项目的结论是：

- 不再把字幕 `clipInfo` 导入方案作为默认修复路径；
- 更保守的 `ImportMedia + AppendToTimeline([subtitle_media_item])` 才是当前主线；
- 如果以后更高版本 Resolve 再次验证 `clipInfo` 路线稳定，再单独建文档记录，不应直接恢复旧结论。

---

## 8. 增量替换为什么会误删

### 8.1 早期最明显的问题

最早的版本会直接清空所有字幕轨道：

```python
for ti in range(1, sub_count + 1):
    items = timeline.GetItemListInTrack("subtitle", ti)
    if items:
        timeline.DeleteClips(items)
```

这当然会导致整个时间线字幕被全部替换。

### 8.2 后来更隐蔽的问题：只删重叠也可能误删

即使后续只删除“与目标时间段重叠”的旧字幕，如果这里的时间段来自“预期位置”而不是“真实导入结果”，仍然会有风险：

- 新字幕真实落点和预期不一致；
- 删除逻辑却按预期区间执行；
- 结果就是误删用户已有字幕。

### 8.3 更隐蔽的问题：同一次导入的新字幕也可能被误删

曾出现过这样的日志特征：

- `Resolve 实际字幕范围` 很短，只覆盖前面一小段；
- 但后面又删掉了很多本应属于新导入字幕的条目。

这说明一个关键事实：

- `AppendToTimeline()` 对字幕返回的对象集合，不一定等于本次新导入的完整字幕集合。

如果脚本仅把这些返回项视为“新字幕排除列表”，那么：

- 前半段新字幕可能被排除；
- 后半段新字幕却仍被当成“旧字幕”命中删除逻辑；
- 最终就会出现“新字幕后半段也没了”的情况。

## 9. 当前项目采用的稳定策略

下面是当前已经沉淀下来的修复思路。

### 9.1 提前锁定音频目标起点

在 `process_minimax_request()` 中：

- 在插入音频前就先计算 `start_frame`；
- 音频和字幕都复用这个 `start_frame`；
- 不在音频插入后再次重新获取“当前时间线位置”。

这样可以避免字幕起点跟着已变化的播放头走偏。

### 9.2 字幕导入前显式恢复播放头

在 `import_srt_to_timeline()` 中：

- 先记录用户原始播放头时间码；
- 再把播放头显式设到音频目标起点；
- 导入结束后恢复原时间码。

这是当前修复链路中非常关键的一步。

### 9.3 保守导入字幕，不走字幕 `clipInfo`

当前项目对字幕使用：

- `ImportMedia([srt_path])`
- `AppendToTimeline([subtitle_media_item])`

原因不是它“完美”，而是它在当前环境下比字幕 `clipInfo` 更稳。

### 9.4 用时间线前后快照识别“本次新字幕全集”

当前实现不会只信 `AppendToTimeline()` 的返回值，而是：

1. 导入前扫描所有字幕轨道；
2. 记录每个 `TimelineItem.GetUniqueId()`；
3. 导入后再次扫描；
4. 通过差分识别这次真正新增的所有字幕项。

这一步是避免误删新字幕后半段的关键。

### 9.5 只在正向补偿时生成 `_positioned.srt`

当前策略是：

- 如果实测发现字幕真实起点晚于目标起点，需要继续向后推，才生成 `_positioned.srt`；
- `_positioned.srt` 是辅助校正文件，不是业务主文件；
- 如果需要负向补偿，不直接用“负时间戳 SRT”去硬修。

### 9.6 只有位置确认正确后才删旧字幕

删除旧字幕的前提变成：

- 已识别本次新字幕全集；
- 已读取新字幕真实起止范围；
- 最终实测起点和目标起点一致。

如果这些条件有任意一项不满足：

- 就跳过旧字幕清理；
- 优先保护用户已有字幕；
- 宁可多留旧字幕，也不冒险误删。

### 9.7 媒体池归档

字幕素材不再直接落在 `Master`：

- 导入时优先放入媒体池 `SRT` 子文件夹；
- 如果 `SRT` 文件夹不存在，则自动创建。

---

## 10. `_positioned.srt` 的定位

`_positioned.srt` 的定位应该明确：

- 它是位置校正过程中的辅助副本；
- 它不是 MiniMax 返回的原始字幕；
- 它也不是用户业务上必须长期保留的主字幕文件；
- 它存在的唯一目的，是在当前位置校正流程里作为兼容宿主行为的中间产物。

如果后续需要继续优化体验，可以考虑：

- 成功导入后自动清理无用的校正副本；
- 或把它们集中放到专门的 cache 目录。

---

## 11. 当前实现的残余风险

虽然当前方案相比之前稳定很多，但仍有几个现实约束需要承认：

### 11.1 Resolve 对字幕导入的内部行为仍不透明

官方脚本文档描述了通用 API 接口，但没有把字幕媒体导入时的全部细节说透，因此仍要依赖实测验证。

### 11.2 宿主行为可能随版本变化

当前结论高度依赖 Resolve `20.2.1` 的实测。

- 升级版本后，字幕导入行为可能又会变化；
- 因此文档里所有“当前稳定策略”都应带版本语境理解。

### 11.3 负向补偿仍是高风险区

当前已经避免直接生成负时间戳 SRT，但如果宿主再次出现异常的前移/后移行为，仍需要继续通过日志验证，而不是再凭推理下结论。

## 12. 当前排查时优先看哪些日志

排查字幕问题时，建议优先关注以下日志：

- `字幕相对时间范围`
- `预期字幕绝对时间范围`
- `Resolve 实际字幕范围`
- `检测到字幕起点偏移`
- `校正后字幕范围`
- `字幕最终起点仍与目标不一致，跳过旧字幕清理`

这些日志分别对应：

- 原始字幕的相对时间；
- 本次目标落点；
- Resolve 实际导入结果；
- 是否进入补偿流程；
- 补偿后结果；
- 是否触发“宁可不删也不误删”的保护逻辑。

### 12.1 代表性实机案例：负向校正导致头部截断

用户曾提供过这样一组关键日志：

```text
预期字幕绝对时间范围：123181 - 124309 帧
Resolve 实际字幕范围：123747 - 124876 帧，共 15 条
检测到字幕起点偏移：-566 帧，准备重新校正导入。
字幕时间已调整，偏移量：-18.89 秒 (-566 帧)
校正后字幕范围：123156 - 123719 帧，共 8 条
字幕最终起点仍与目标不一致（目标 123181，实际 123156），跳过旧字幕清理以避免误删。
```

这组日志很重要，因为它把问题链条暴露得非常完整：

- 第一次导入时，Resolve 实际落点明显晚于目标起点。
- 脚本尝试做负向补偿，把整份字幕整体前移。
- 前移后，字幕条数从 `15` 条变成 `8` 条，说明头部内容已经被截断。
- 最终起点依然没有精确对齐，因此不能继续执行旧字幕清理。

这也是当前文档明确反对“把负向时间补偿直接写回 SRT”作为通用解的核心依据。

### 12.2 `Ctrl + Z` 反馈证明这不是显示问题

用户还提供过一个非常关键的实机现象：

- 新字幕导入后，后方已有字幕消失。
- 随后执行 `Ctrl + Z` 撤销，原来的字幕会重新出现。
- 重新出现后的字幕位置没有变化，只是恢复了被删除状态。

这个现象说明两件事：

- 问题不是简单的界面刷新异常，也不是 Resolve 只显示了一部分字幕。
- 删除逻辑确实对时间线现有字幕执行了真实删除操作。

因此，后续修复策略必须把“删除旧字幕”视为高风险操作，并且只能建立在“真实导入范围已经确认正确”的前提上。

### 12.3 `_positioned.srt` 为什么保留

在上述案例之后，项目里保留 `_positioned.srt` 的原因也需要明确：

- 它不是重复产物，而是“正向补偿时的中间校正副本”。
- 保留原始 `.srt`，可以避免把 MiniMax 原始结果直接污染掉。
- 一旦校正失败，也能从原始 `.srt` 回看问题，而不是只剩一份被修改过的字幕文件。

因此，当前方案不是简单地“多生成了一份冗余字幕”，而是在可追踪性和宿主兼容性之间做的折中。

---

## 13. 当前已经淘汰的做法

以下做法不应再写成当前项目的推荐方案：

- 把字幕优先改成 `clipInfo + recordFrame` 精确插入。
- 无条件按 `insert_position_frame` 改写整份 SRT 时间戳。
- 只依赖 `AppendToTimeline()` 的返回值识别本次新增字幕。
- 在未确认真实导入位置前，先按预期时间段删除旧字幕。
- 把字幕问题理解成“纯数学偏移问题”。

---

## 14. 当前结论

在本项目和 DaVinci Resolve `20.2.1` 的组合下，字幕问题不是单一的“时间偏移 bug”，而是以下因素叠加造成的：

- 宿主的字幕导入行为与音频插入行为不同；
- 当前播放头状态会影响字幕导入结果；
- `AppendToTimeline()` 返回值不一定覆盖全部新字幕；
- 删除旧字幕的时机如果控制不好，就会误删用户内容；
- 某些理论上更“精确”的字幕插入方式，在当前环境下反而不稳定。

因此，当前最可靠的经验不是更激进地推导时间戳，而是：

- 先锁定音频目标起点；
- 导入前恢复播放头；
- 导入后读取真实落点；
- 用时间线差分识别完整新字幕；
- 只在位置确认正确后才删除旧字幕；
- 在所有不确定场景下优先保护用户已有字幕。
